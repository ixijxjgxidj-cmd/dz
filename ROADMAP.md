# 地震震相拾取系统 — 优化实施路线图

> 本文档是对原始方案的批判性优化版本。核心思想：**先想清楚"分从哪来"，再写工程代码。**
> 工程骨架（FastAPI / Docker / 压测）照做即稳，真正决定名次的是三个"胜负手"，已在下方用 🎯 标出。

---

## 0. 三条被确认的关键约束（决定了整套设计）

| 约束 | 现状 | 设计应对 |
|---|---|---|
| 能否拿到可微调的标注数据 | **还不确定** | 微调设计成"数据一到位即可开跑"的可插拔主线；但系统在**零微调**下也要能完整跑分。不赌数据一定存在。 |
| 官方 API 接口规范 | **还没拿到** | 请求/响应全部走**适配层（adapter）**。业务内核只认内部数据结构；官方规范一到，只改 adapter，不动内核。 |
| 部署环境 | 前期本地 GPU，提交时云 GPU | 代码 **GPU/CPU 通吃**，`device` 可配置（`auto`/`cpu`/`cuda`）。推理量小，CPU 也能扛，作为可用性兜底。 |

## 1. 🎯 三个胜负手（优先级最高，先于一切工程封装）

### 胜负手 A — 微调把 P 波精度压进 0.1s
- P 波满分阈值 0.1s，在 100Hz 下只有 **10 个采样点**。SeisBench 预训练权重在异域数据上残差常漂到 0.1~0.3s，**只靠预训练几乎不可能冲高分**。
- 因此：**微调不是加分项，是主线**（只要数据允许）。PhaseNet 模型很小，RTX4060 8GB 微调绰绰有余。
- 数据未定，所以先把训练脚本接口与数据管线预留好，数据一到立即开跑。

### 胜负手 B — 绝对时间对齐绝对不能错
- 到时 = `trace.stats.starttime + sample_index / sampling_rate`。
- 高危陷阱：重采样后 starttime/采样率未同步、UTC/时区、mseed 多段 gap、整数/浮点误差。
- **一旦系统性偏移，模型再好也全盘皆输，而且很难察觉。**
- 对策：时间对齐单独成模块 + **单元测试用已知答案反复验证**（本仓库已实现并通过测试）。

### 胜负手 C — 用本地评分脚本驱动阈值优化，且偏向"高精确率"
- 数量误差惩罚很重（超 5% 每多 1 个扣 0.5 分）→ **宁可漏检，不可多检**。
- 检测阈值（probability threshold、trigger_on/off）应由**本地评分脚本反向搜索**得出，不能拍脑袋。
- 评分脚本是整个项目的"体温计"，因此**提前到最小闭环之后立刻实现**（本仓库已实现并通过测试）。

---

## 2. 相对原方案的关键改动

1. **执行顺序调整**：本地评分脚本从第 4 步提前到"数据+推理最小闭环"之后立即做。理由：没有体温计，后面所有调优都是盲调。
2. **微调提优先级**：从"可扩展加分项"提升为主线任务（数据允许时）。
3. **EQTransformer 集成先搁置**：团队非 AI 背景 + 算力有限，先把 PhaseNet 单模型（微调+阈值调优）打透。集成留作决赛专家评分（20%）的展示，而非主线。
4. **部署架构**：推理服务优先考虑**稳定云 CPU 主机**保可用性，GPU 只用于训练；本地 GPU 服务作为开发环境。避免"家用网络跑高可用 API"的短板。
5. **并发陷阱显式处理**：单 worker + 推理走线程池/串行队列，避免多 worker 各加载一份模型撑爆显存；async 路由不直接跑阻塞推理。
6. **超时策略修正**：GPU 推理启动后难以中途 kill，改为**限制输入波形时长上界**间接保证不超时，而非指望强杀。
7. **预测↔真值匹配算法**：评分脚本明确采用"同类型 + 容差窗口内 + 时间最近"的贪心匹配（本仓库已实现）。
8. **加速度/速度域差异**：预留"积分成速度再送模型"的开关，并在微调阶段可混入加速度样本（实验对比项）。

## 4. 决赛专家评分（20%）加分素材（边做边攒）
- 阈值/参数选择的**实验依据表**（本地分数对比）——最有说服力的技术文档素材。
- 固定随机种子保证可复现；模型权重版本管理；验证集与调参集分离防过拟合。
- 关键设计决策注释：为什么选 PhaseNet、去重逻辑原理、时间对齐为何单独测试。
